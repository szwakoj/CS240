    Hashed Malware: Making a Malware Hash Map

    Due Date: 5/3/26

    Description

    In this homework, you will create a system to similar ones that back malware detection software like Norton and Windows Defender, a hashed lookup table of known malicious software. This directly is an implementation of the lesson on hash maps and takes advantage of its fast lookup times. The situation is this:

    • On an operating system, any executable from the open web needs to be verified to be safe otherwise the system is open for hacking (...which is bad).
      • Call it \(files\), an \(n\) long array of file names
      • Some number, \(m\) where \(0 \le m \le n\), of the \(files\) are malicious, forming \(malware\), \(malware \subseteq files \quad \text{where } malware = \{f \in files \mid f \text{ is malicious}\}\)
    • We need to detect these malicious files before a process or user accidentally runs them
      • This leaves the following computational problem:
        • Input: An array \(files\) of \(n\) file names \([f_0, f_1, \ldots, f_{n-1}]\), where \(n > 0\), and some number \(m\) where \(0 \leq m \leq n\) of the files are malicious
        • Output: A subset \(malware \subseteq files\) such that \(malware = {f \in files \mid f \text{ is malicious}}\)
    • Detecting malware is extremely hard just by looking at machine code, impossible if you don't know exactly what you are looking for. Normal machine code is as unreadable as malicious machine code, plus they are identical in structure (however machine code is almost unique to the software it runs).
    • So usually preemptive detection of malware on an OS is done via comparison to known malware samples that are collected in the wild by volunteers and companies
    • There are repositories that are updated in real time where people will report malware they find for researchers, giving the executable files that are the cause to aid in this process of detection.
    • We can download these repositories and use it to compare the source code of the executables in the system! Problem solved!
      • Lets call the list of known malware \(known\), an \(k\) long array of file names
      • We could just check each file in \(files\) and compare it to each \(known\), adding any confirmations to \(malware\) for later removal:
        • Procedure:
          1. \(malware \gets [\ \ \ ]\)
          2. \(\text{for } i \gets 0 \text{ to } n-1 \text{ do:}\)
            1. \(\text{for } j \gets 0 \text{ to } k-1 \text{ do:}\)
              1. \(\text{if } files[i] = known[j] \text{ then:}\)
                1. \(malware.append({files[i]})\)
              2. \(\text{end if}\)
            2. \(\text{end for}\)
          3. \(\text{end for}\)
          4. \(\text{return } malware\)
    • This runs in \(O(n \cdot k)\) time, for every file in \(files\) we scan all \(k\) entries in \(known\). Both new software and malware are released every hour making the growth of \(n\) and \(k\) very real and fast.
      • Estimated total of malware samples known, \(k \gt 10^9\) (source)
      • \(k\) grows at about 6,500 new malware samples a week (source
      • Realistic approximates for executables on a modern system can reach in the tens of thousands, \(n \gt 10^4\)
      • With memory and people thirst for more software products not having an end in sight, the speed of this growth would be hard to measure as concretely, however we can be sure it is growing
      • With both \(n\) and \(k\) growing, and already the numbers being so large, it would take days for a computer to scan its contents fully
      • We need something better
    • One such malware repository is Malwarebazaar, check them out

      • Think of this like a Zoo's lion exhibit. You can look, see where it came from, and maybe listen to a trained professional explain how they caught it in the cage. However, YOU DO NOT BRING IT HOME WITH YOU.
      • Windows people are especially vulnerable nowadays with Windows adding Linux support AND it being targeted by almost all of the file types on there

    • If you look at the list of recent submissions you will notice a field called "SHA256 hash" attached to each malware sample
    • This is a unique identifier that is generated by a hashing function as described in the lesson
    • Instead of searching each file and comparing it to each element in \(known\), we can instead use a hashing algorithm store a hashmap of \(known\) using a really good hashing algorithm like SHA256 and then hash incoming files to see if there is an entry already:
      • Procedure:
        1. \(knownMap \gets \{\}\)
        2. \(\text{for } j \gets 0 \text{ to } k-1 \text{ do:}\)
          1. \(knownMap.\text{insert}(known[j])\)
        3. \(\text{end for}\)
        4. \(malware \gets [\ \ \ ]\)
        5. \(\text{for } i \gets 0 \text{ to } n-1 \text{ do:}\)
          1. \(\text{if } knownMap.contains(files[i]) \text{ then:}\)
            1. \(malware.append({files[i]})\)
          2. \(\text{end if}\)
        6. \(\text{end for}\)
        7. \(\text{return } malware\)
    • Building \(knownMap\) costs \(O(k)\), and each lookup \(knownMap.contains(files[i])\) costs \(O(1)\) on average, giving an overall runtime of \(O(n + k)\), a significant improvement over the \(O(n \cdot k)\) linear scan
    • If we precompute \(knownMap\) then we can pay that computational cost upfront, or, in the case of MalwareBazaar, offload to other people, that \(O(k)\) becomes loading a file from memory of hashes rather than recalculating all previous hashes
      • This also means we don't need to keep a folder of the same bad software we are trying to avoid!

    Summation Hash

    This is a simple algorithm where you take in the contents of an executable file by the byte and add them together, ending with a modulus by the max table size:

    • Algorithm: Summation Hash
      • Input: An array \(bytes\) of \(n\) bytes \([b_0, b_1, \ldots, b_{n-1}]\) representing the contents of an executable file, and an integer \(tableSize\) representing the maximum table size
      • Output: An integer index \(h\) where \(0 \leq h < tableSize\)
      • Procedure:
        1. \(sum \gets 0\)
        2. \(\text{for } i \gets 0 \text{ to } n-1 \text{ do:}\)
          1. \(sum \gets sum + bytes[i]\)
        3. \(\text{end for}\)
        4. \(\text{return } sum \mod tableSize\)

    Polynomial Rolling Hash

    Similar to the Summation Hash, you are summing the bytes together, except this time you are also scaling each byte of data by a prime number raised to the power of the index of the current byte:

    • Algorithm: Polynomial Rolling Hash
      • Input: An array \(bytes\) of \(n\) bytes \([b_0, b_1, \ldots, b_{n-1}]\) representing the contents of an executable file, an integer \(tableSize\) representing the maximum table size, and a prime \(p\)
      • Output: An integer index \(h\) where \(0 \leq h < tableSize\)
      • Procedure:
        1. \(sum \gets 0\)
        2. \(\text{for } i \gets 0 \text{ to } n-1 \text{ do:}\)
          1. \(sum \gets sum + bytes[i] \cdot p^i\)
        3. \(\text{end for}\)
        4. \(\text{return } sum \mod tableSize\)

    djb2

    djb2 was invented by Dan Bernstein in 1991 on a C programming, and after that it floated around on the internet for a while and gained popularity since it was easy to implement and was pretty good. Most mysteriously of all is its magic number 33 which is the main means of encoding and does better than other number. Which is weird!

    • Algorithm: djb2
      • Input: An array \(bytes\) of \(n\) bytes \([b_0, b_1, \ldots, b_{n-1}]\) representing the contents of an executable file, and an integer \(tableSize\) representing the maximum table size
      • Output: An integer index \(h\) where \(0 \leq h < tableSize\)
      • Procedure:
        1. \(hash \gets 5381\)
        2. \(\text{for } i \gets 0 \text{ to } n-1 \text{ do:}\)
          1. \(hash \gets hash \cdot 33 + bytes[i]\)
        3. \(\text{end for}\)
        4. \(\text{return } hash \mod tableSize\)

    FNV-1a (Fowler–Noll–Vo)

    FNV is an algorithm created by Glenn Fowler, Landon Curt Noll, and Kiem-Phong Vo in 1991 also, however it was submitted to the IEEE rather than a C forum. It works by multiplying the hash by a prime and XORing the bytes of the file for each byte. This repeated prime multiplication makes each input generate a fairly unique hash.

    Use the constants at the top that I provide.

    • Algorithm: FNV-1a (Fowler–Noll–Vo)
      • Input: An array \(bytes\) of \(n\) bytes \([b_0, b_1, \ldots, b_{n-1}]\) representing the contents of an executable file, and an integer \(tableSize\) representing the maximum table size
      • Output: An integer index \(h\) where \(0 \leq h < tableSize\)
      • Procedure:
        1. \(hash \gets \text{offset-basis}\)
        2. \(\text{for } i \gets 0 \text{ to } n-1 \text{ do:}\)
          1. \(hash \gets hash \oplus bytes[i]\)
          2. \(hash \gets hash \cdot \text{fnv-prime}\)
        3. \(\text{end for}\)
        4. \(\text{return } hash \mod tableSize\)
